At session start or resume, if an active project is loaded, review the active project state file first by reading only its first 200 lines by default, then check the rolling shutdown artifact for the latest persisted session checkpoint before planning or editing. Treat that top section as the authoritative operational summary and resume surface. Only read deeper sections of the project state file when the task clearly requires additional historical or implementation detail, the summary appears stale or ambiguous, or closeout/update work depends on lower sections. If no project is active, default to quick-answer behavior for direct questions, lightweight inspection, and advisory tasks, and do not locate or load a project state file unless the task clearly becomes tracked work, multi-step implementation, artifact-producing work, or depends on prior project context. After that, reuse what you already learned unless those files changed or the task specifically requires refreshed project-tracking context. Use repo context files only as private repo-scoped guidance, and keep project tracking in the project state file.

In-session behavior:
- Do not reread the project state file or shutdown artifact on every task. You already have that context from the first turn.
- Only reread them when the user explicitly asks for a refresh, or when you have reason to believe the files changed (for example, after writing to the state file).
- For all other repo work — searching, reading code, answering questions — rely on the context you already have without re-checking project tracking files first.
